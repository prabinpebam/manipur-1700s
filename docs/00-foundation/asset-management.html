Asset Management

How images, references, fonts, and audio move from a messy inbox into an organized library, and finally onto the documentation pages that use them.

TL;DR

Dump anything into _intake/ without thinking. When you have a moment, rename it, move it into the right docs/assets/… folder, and reference it from a docs page. Intake stays messy and temporary; the asset library stays clean and is the only place docs point at.

The flow

  1. 1 · Drop into _intake/

    A single repo-root inbox. Any format, any name, subfolders optional. No rules - the point is zero friction so nothing gets lost.

  2. 2 · Curate into docs/assets/…

    Rename to the convention, web-optimize, and move into the matching library folder. One subject per file.

  3. 3 · Reference from a docs page

    Add an <img> inside a slate-figure with real alt text and a caption. Only ever point at docs/assets/…, never at _intake/.

  4. 4 · Clear the inbox

    Once a file has moved out, delete it from _intake/. The inbox should trend back toward empty.

Naming convention

One predictable pattern

<area>-<subject>-<short-desc>-<nn>.<ext> - lowercase, hyphenated, two-digit sequence. Areas: sdv, jp, art, font, name.

How a page points at an asset

Paths are relative to the page that includes them. Subfolder pages (like the ones in 15-benchmarking/ or 40-art-audio/) reach the library with ../assets/…; top-level pages use assets/….

What not to commit

Keep the repo light

Heavy source files (PSD, AI, RAW, uncompressed audio/video) are git-ignored. Export a web-friendly version into docs/assets/… and keep the source elsewhere. Empty library folders are kept alive with a .gitkeep file.
